PHASE 2 — SECRETS AND INTEGRATIONS REBUILD
==================================================

This is the highest priority. Fix this first.

CURRENT PROBLEM TO ELIMINATE
- Secrets must never travel through the client unnecessarily.
- Secrets must never be inserted into system prompts.
- Secrets must never be sent to the LLM as raw values.
- Secrets must never be returned to the frontend after save.
- Secrets must never be stored unencrypted in general-purpose JSON fields.

TARGET ARCHITECTURE
Replace the current integrations flow with a secure server-side vault pattern.

REQUIRED CHANGES

1. CREATE A SECURE INTEGRATIONS STORAGE MODEL
- Create a dedicated server-side storage model for user/provider credentials.
- Do not store raw provider secrets in generic user profile JSON or DNA blobs.
- Store each integration as a separate record with fields like:
  - id
  - userId
  - provider
  - encryptedSecretPayload
  - metadata (safe non-secret fields only)
  - createdAt
  - updatedAt
  - lastValidatedAt
  - status
- Encrypt secret payloads before database storage using application-level encryption.
- Ensure encryption utilities are consistently used.

2. SEPARATE SAFE METADATA FROM SECRETS
- Safe metadata may include:
  - provider name
  - connected / not connected
  - partial identifier if needed
  - validation status
  - capabilities supported by provider
- Never expose secret values to the client.

3. CHANGE API CONTRACTS
- Frontend endpoints for integrations must only receive and render safe metadata.
- Saving credentials should be write-only from the client perspective.
- Reading integrations must return only safe metadata and capability flags.
- Remove any endpoint behavior that returns raw keys after save.

4. REMOVE SECRETS FROM PROMPT GENERATION
- Refactor system prompt generation so the model only receives capability-level information.
- Example of allowed prompt content:
  - “The user has Stripe connected”
  - “The project can use OpenAI via server-side integration”
  - “Firebase is available through secure backend configuration”
- Example of forbidden prompt content:
  - actual API keys
  - tokens
  - URIs
  - auth secrets
  - service account credentials

5. REMOVE SECRETS FROM MESSAGE PAYLOADS
- Chat/message requests from frontend must not include raw integration secrets.
- Replace this with capability references or integration IDs where appropriate.

6. SERVER-SIDE TOOL EXECUTION ONLY
- If the model needs to use a connected service, route it through secure backend tools/adapters.
- Never generate code that requires embedding server secrets in frontend runtime.
- Prefer server proxy patterns where applicable.

7. LOGGING REDACTION
- Add or improve redaction so logs, errors, and telemetry never expose:
  - API keys
  - bearer tokens
  - DB URLs
  - webhook secrets
  - auth headers
  - provider secrets
- Redact before writing logs, telemetry, traces, or prompt history.

8. TESTS FOR THIS PHASE
Add tests for:
- saving integrations encrypts values
- reading integrations never returns raw secrets
- prompts never contain secret strings
- message APIs reject raw secret transport patterns
- logs redact known secret patterns

PHASE 3 — AUTHORIZATION AND PROJECT ACCESS MODEL
==================================================

This is also critical.

CURRENT PROBLEM TO ELIMINATE
- Anonymous / ownerless projects must not become effectively unprotected.
- Authorization logic must be unified and enforceable everywhere.

TARGET AUTH MODEL
Introduce explicit access modes:
- private owner project
- guest session project
- public read-only share
- team/shared project if applicable

REQUIRED CHANGES

1. UNIFY PROJECT ACCESS MIDDLEWARE
Implement clear middleware or service layers such as:
- requireAuth
- loadProject
- requireProjectReadAccess
- requireProjectWriteAccess
- requireProjectOwner
- requireProjectAdminAccess if needed

Do not rely on scattered ad hoc checks across many routes.

2. FIX OWNERLESS / ANONYMOUS PROJECT LOGIC
If guest projects are allowed:
- bind them to a signed guest session token or durable guest identity
- explicitly track access scope
- do not allow arbitrary access based only on project id
- define expiration / upgrade flow
- define transfer flow if guest becomes registered user

If guest projects are not needed:
- remove ownerless project write flows entirely

3. DEFINE SHARE / PUBLIC ACCESS EXPLICITLY
Public access must be intentional and scoped.
- read-only public share should never imply write access
- write endpoints must reject public/share tokens unless explicitly allowed
- share permissions should be modeled, not guessed

4. ROUTE-BY-ROUTE HARDENING
For every route under project scope:
- verify required access level
- verify read vs write distinction
- verify share/public behavior
- verify anonymous access is either safely allowed or denied

5. TESTS FOR THIS PHASE
Add an authorization matrix test suite covering:
- owner can read/write/delete
- non-owner cannot mutate
- guest can access only own guest project if applicable
- public share can read only if intended
- ownerless project ids alone do not grant access
- route coverage for all project mutation endpoints
