PHASE 2 — STABILIZE AND SECURE THE PLATFORM
==================================================

Before major product upgrades, secure the current system.

You must audit and harden:
- terminal websocket
- collab websocket
- auth middleware
- preview/share endpoints
- deploy endpoints
- GitHub sync flows
- proxy routes
- secrets handling
- CORS policy
- session handling
- rate limiting
- user/project authorization checks
- SQL execution paths
- any path that allows code execution or external requests

Security requirements:
1. Every websocket route must require authentication and authorization.
2. Every project-scoped route must verify ownership or access permission.
3. Terminal access must not expose unrestricted shell execution.
4. If terminal access cannot be made safe enough, isolate it or disable it behind a feature flag.
5. CORS must be explicit, environment-aware, and locked down in production.
6. Secrets must never leak to frontend responses, logs, previews, or public share pages.
7. Share endpoints must expose only what is intended.
8. Add consistent audit logging for risky operations.
9. Add centralized input validation and output sanitization where needed.
10. Add secure defaults for cookies, sessions, auth refresh, and CSRF-sensitive flows.

Create:
- /docs/security/threat-model.md
- /docs/security/hardening-checklist.md

Then implement the hardening.
