יש כאן חוסר הבנה מהותי של המוצר שאני רוצה.

אני לא רוצה מערכת מסובכת מדי עם יותר מדי פונקציות נפרדות, יותר מדי כלים צדדיים, יותר מדי מסכים, ויותר מדי זרימות פנימיות.

אני רוצה מוצר פשוט מאוד להבנה מצד המשתמש, אבל חזק מאוד בביצוע.

החזון הנכון של המוצר:
המשתמש נכנס למערכת, כותב בצ'אט מה הוא רוצה לבנות, והמערכת עושה עבורו את כל השאר מקצה לקצה.

זה כולל גם מצב של:
- בנייה של פרויקט חדש מאפס
- עדכון פרויקט קיים
- תיקון באגים בפרויקט קיים
- תיקון קבצים קיימים
- שינוי קוד קיים לפי בקשה בצ'אט
- הוספת פיצ'רים לקוד קיים
- refactor ממוקד כשצריך
- שמירה על מה שכבר קיים בלי להרוס אותו

ה-flow הנכון צריך להיות:

1. המשתמש כותב בצ'אט מה הוא רוצה
זה יכול להיות אחד מאלה:
- "תבנה לי אפליקציה חדשה"
- "תוסיף לי מסך התחברות"
- "תתקן לי באג"
- "הפונקציה הזו לא עובדת"
- "תשנה לי את הניווט"
- "תוסיף חיבור ל-GitHub"
- "תעדכן את ה-deploy"
- "תשפר את הטפסים"
- "תסדר את הקוד הקיים"

2. המערכת צריכה להבין את סוג הבקשה
המערכת חייבת לדעת להבדיל אוטומטית בין:
A. build new
B. update existing
C. fix bug in existing code
D. extend existing project
E. deploy / publish / sync operation

3. במצב של קוד קיים
אם כבר יש פרויקט קיים, המערכת חייבת:
- לקרוא את מצב הפרויקט הנוכחי
- להבין את הקבצים הקיימים
- להבין מה כבר בנוי
- להבין מה המשתמש ביקש לשנות
- לבצע שינוי ממוקד
- לא לדרוס את כל הפרויקט מחדש בלי צורך
- לא למחוק חלקים עובדים
- לא להתחיל rebuild מלא אם המשתמש ביקש רק תיקון

4. זה קריטי מאוד:
המוצר הוא לא רק builder מאפס.
הוא גם editor + fixer + maintainer דרך chat.

כלומר:
המשתמש צריך להיות מסוגל לדבר עם המערכת לאורך כל חיי הפרויקט:
- לבנות
- לתקן
- לשפר
- להוסיף
- להסיר
- לפרוס
- לעדכן גרסאות
- לדחוף קוד
- לטפל בבאגים

5. מה המערכת צריכה לדעת לעשות בפועל
אחרי הצ'אט, המערכת צריכה להיות מסוגלת:
- לייצר קבצים חדשים
- לעדכן קבצים קיימים
- לתקן לוגיקה קיימת
- לשמור על consistency בין הקבצים
- לעדכן preview
- לשמור state
- לעדכן GitHub
- לבצע push
- להכין deployment
- לפרוס לענן
- להחזיר למשתמש תוצאה usable
- ואז להמשיך לקבל ממנו הוראות נוספות בצ'אט

6. חוויית המוצר צריכה להיות פשוטה
אני לא רוצה שהמשתמש ירגיש שהוא עובד מול אוסף כלים.
אני רוצה שהוא ירגיש שהוא עובד מול עוזר בנייה חכם שיודע:
- לבנות
- לתקן
- לערוך
- לייצב
- לפרוס

הכול מאותו צ'אט.

7. העיקרון המרכזי
Chat-centered end-to-end software builder.

המשמעות:
הצ'אט הוא לא רק ליצירה ראשונית.
הצ'אט הוא גם:
- תיקון באגים
- תחזוקה
- שינויים
- שיפורים
- refactors ממוקדים
- sync + deploy

8. מה אני לא רוצה
אני לא רוצה מצב שבו:
- המערכת יודעת רק לייצר משהו חדש
- אבל לא יודעת לתקן קיים
- או שבכל שינוי קטן היא בונה הכול מחדש
- או שהיא לא מבינה dependency על קבצים קיימים
- או שהיא מוחקת עבודה קיימת בגלל שינוי מקומי
- או שהיא מטפלת ב-Git / deploy / preview כאילו הם כלים צדדיים

9. מה אני כן רוצה
אני רוצה flow אחד ברור:
user chats -> system understands intent -> system reads current project state ->
system builds/fixes/updates code -> system refreshes preview ->
system syncs GitHub -> system deploys/publishes ->
user continues through chat

10. אני רוצה שתעצב מחדש את המוצר סביב 5 מצבי עבודה ברורים בלבד
- Create
- Edit
- Fix
- Sync
- Deploy

אבל כל אלה צריכים להרגיש למשתמש כחלק מצ'אט אחד פשוט,
לא כעולם שלם של פיצ'רים מפוזרים.

11. מה צריך להשתנות עכשיו במערכת
אני רוצה שתעבור על המוצר ותיישר אותו לפי ההבנה הזאת.

תבדוק ותשנה:
A. שהצ'אט הוא המרכז
B. שהמערכת יודעת לעבוד גם על קוד קיים, לא רק על קוד חדש
C. שכל flow של תיקון באג עובר דרך קריאת state קיים + שינוי ממוקד
D. ש-preview, GitHub, ו-deploy הם חלק מה-flow הראשי
E. שאין rebuild מלא כשלא צריך
F. שאין complexity מיותרת שנחשפת למשתמש
G. שהממשק תומך במוצר כעוזר בנייה מלא לאורך זמן, לא רק בתחילת הפרויקט

12. מה אני רוצה ממך עכשיו
אני רוצה ממך:
1. להסביר מה במוצר הנוכחי עדיין מפספס את ההבנה הזאת
2. להגדיר את user flow הנכון
3. להגדיר את system flow הנכון
4. לציין מה בממשק מיותר או מפוזר מדי
5. לציין מה צריך להיות מוסתר או פנימי
6. לציין מה צריך להיות ליבת הממשק
7. ואם צריך — גם לתקן בפועל קבצים כדי ליישר את המוצר לחזון הזה

13. הדבר הכי חשוב
המערכת חייבת להבין שהיא לא רק:
"צ'אט שמייצר קוד"

היא צריכה להיות:
"צ'אט שמנהל את כל חיי הפרויקט"

כלומר:
- בונה
- משנה
- מתקן
- משדרג
- מסנכרן
- פורס

14. תוצאה שאני מצפה ממך
אני רוצה שהמוצר ירגיש כמו מערכת אחת פשוטה:
המשתמש מדבר,
המערכת מבצעת,
המשתמש ממשיך לדבר,
והמערכת ממשיכה לתקן/לעדכן/לפרוס.

זה החזון.
תתאים את המערכת אליו.
